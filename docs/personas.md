# Personas

## Primary Personas

### Caller: 住客 / 来电预订人

- 目标：通过电话快速获取可选房型、价格与可订状态
- 提供信息：期望房型、姓名、手机号
- 需避免的情况：AI 声明有房但最终无法完成预订，或反复追问无关信息
- 成功标准：在 1 通电话内完成咨询、确认与预订

### AI Receptionist: AI 酒店前台

- 目标：像前台一样自然对话，基于内部库存与预订接口做事实判断
- 核心职责：
  - 先查房型与实时余量，再回复可订情况
  - 无房时推荐仍有库存的替代房型
  - 收集预订所需信息：房型、姓名、手机号
  - 调用预订接口并将结构化结果翻译成自然语言
- 禁止行为：
  - 编造库存
  - 在接口未返回成功前承诺预订完成
  - 忽略无房状态继续下单

### Hotel Operator: 酒店运营 / 演示人员

- 目标：在网页中查看当前房型余量与最新预订结果，用于现场演示与排障
- 关心信息：
  - 各房型剩余数量
  - 预订状态
  - 预订人姓名与手机号
- 成功标准：无需查看日志即可理解 demo 当前状态

### Builder: 开发者 / 维护者

- 目标：在不破坏 WebRTC 与会话边界的前提下，将酒店预订能力接入为清晰的垂直场景
- 关心信息：
  - 内部服务边界是否清晰
  - tool contract 是否稳定
  - 文档、协议、测试是否同步

## Core Journey

1. 来电用户询问可预订房型
2. AI 前台调用库存能力，返回仍可预订的房型与数量
3. 用户选择房型，提供姓名与手机号
4. AI 前台调用预订接口
5. 系统返回 `confirmed`、`sold_out` 或 `failed`
6. AI 前台将结果告知用户；无房时推荐仍有库存的替代房型

## Demo Success Signal

- 用户能通过语音自然完成「问房 -> 选房 -> 留资 -> 预订」
- 酒店运营能在网页中同步看到库存变化与预订记录
- AI 全程以工具返回结果为准，不出现口头状态与系统状态不一致的情况
