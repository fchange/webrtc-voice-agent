# Personas

## Primary Personas

### Caller: 住客 / 来电预订人

- 目标: 通过电话快速知道还有什么房型、多少钱、能不能订到
- 提供信息: 想要的房型、姓名、手机号
- 不希望发生的事: AI 说有房但最后订不上，或者反复追问无关信息
- 成功标准: 在 1 通电话里完成咨询、确认和预订结果

### AI Receptionist: AI 酒店前台

- 目标: 像前台一样自然对话，同时严格基于内部库存与预订接口做事实判断
- 核心职责:
  - 先查房型和实时余量，再回复可订情况
  - 在无房时主动推荐仍有库存的替代房型
  - 收集预订所需最小信息: 房型、姓名、手机号
  - 调用预订接口并把结构化结果翻译成自然语言
- 不能做的事:
  - 编造库存
  - 在接口未返回成功前承诺“已经订好”
  - 忽略无房状态继续下单

### Hotel Operator: 酒店运营 / 演示人员

- 目标: 在网页里看到当前房型余量和最新预订结果，方便现场演示和排障
- 关心信息:
  - 各房型剩余数量
  - 预订状态
  - 预订人姓名和手机号
- 成功标准: 不看日志也能理解 demo 当前状态

### Builder: 开发者 / 维护者

- 目标: 在不破坏 WebRTC 与会话边界的前提下，把酒店预订能力接入成一个清晰的垂直场景
- 关心信息:
  - 内部服务边界是否清楚
  - tool contract 是否稳定
  - 文档、协议、测试是否同步

## Core Journey

1. 来电用户询问“还有什么房型可以订”
2. AI 前台调用库存能力，返回仍可预订的房型和数量
3. 用户选择房型，并提供姓名和手机号
4. AI 前台调用预订接口
5. 系统返回 `confirmed`、`sold_out` 或 `failed`
6. AI 前台把结果告知用户；若无房，则推荐仍有库存的替代房型

## Demo Success Signal

- 用户能自然通过语音完成“问房 -> 选房 -> 留资 -> 预订”
- 酒店运营能在网页里同时看到库存变化和预订记录
- AI 全程以工具返回结果为准，不出现“口头有房、系统无房”的状态漂移
